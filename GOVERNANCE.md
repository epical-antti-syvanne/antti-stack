# Governance

Antti Stack uses lightweight maintainer governance until the ecosystem becomes complicated enough to generate its own weather system.

## Principles

- Spec over vibes.
- Signal over ceremony.
- Tests over confidence.
- Mock systems, not people.
- Partner-safe satire over vendor war crimes written in Markdown.
- Useful output over performative cleverness.

## Decision Making

Small changes can be merged by maintainers when they are clear, tested, and aligned with the project voice.

Larger changes should have an issue or TODO entry first, especially when they affect:

- public CLI behavior
- JSON response contracts
- memory storage
- MCP/API tools
- safety boundaries
- README positioning
- vendor satire level

## Maintainer Duties

Maintainers should:

- protect the voice
- keep the tool useful
- ask for tests when behavior changes
- prevent corporate fog from re-entering under a new job title
- avoid turning every decision into a framework

## Community Standard

Contributors are welcome when they improve the work, reduce banality, or add a useful form of absurdity.

Contributions that attack people, expose private data, or create vendor flamewars will be closed with as little ceremony as possible.
